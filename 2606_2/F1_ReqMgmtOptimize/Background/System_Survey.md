# 现有系统调研 (System Survey)

## 1. 原系统核心逻辑 (Original System Core Logic)

当前系统已经具备原始需求管理的基础能力，原始需求在创建或编辑时已有“优先级”字段，优先级值以字典形式维护，当前主要使用 P0 / P1 / P2 / P3 或对应中文标签进行展示与筛选。

原始需求的主要处理入口在“原始需求池”，相关角色可以查看需求基础信息、编辑优先级、推进后续处理。系统中还存在“优先级视图”，用于按优先级分组展示需求，但当前优先级主要是人工录入或人工调整，尚未形成基于业务规则和 AI 的自动判定链路。

从业务上看，优先级是原始需求进入评估和后续处理时的重要决策信息，但当前缺少统一、可维护、自动化的定级机制，因此容易出现以下问题：

- 不同角色对同类需求的优先级判断口径不一致。
- 依赖人工判定，效率较低。
- 规则知识停留在文本或经验层，未被系统消费。
- 难以复盘 AI 或人工判定的依据，缺少可解释性和审计信息。

## 2. 本次需求影响的现有页面 (Impacted Existing Pages)

### 2.1 原始需求池

- 这是本次需求的核心受影响页面。
- 当前页面不是一个独立站点式页面，而是挂载在 `MainLayout` 内容区中的业务页。
- 页面结构已经比较明确，主要包括：
  - 顶部工具栏：标题、说明、搜索框、新建需求按钮；
  - 筛选栏：折叠/展开模式切换、筛选条件入口；
  - 两种主体展示模式：
    - 折叠模式：`compact-list` 卡片列表 + 就地展开详情；
    - 展开模式：`table-wrap` 大表格；
  - 详情内容通过 Accordion 就地展开，而不是独立详情页跳转；
  - 拆分能力通过 `splitDialog` 完成，是当前页面的重要交互中心。
- 因此本期 AI 判级能力必须嵌入这套既有结构中，而不是重做原始需求池页面骨架。
- 更合理的承载方式是：
  - 在现有优先级字段附近增加 AI 判定来源、规则版本、判定摘要入口；
  - 在就地展开详情区增加“AI 判级卡片 / 判定记录 / 重新判级”；
  - 在不打断当前拆分流程的前提下补充 AI 判级信息。

### 2.2 原始需求详情抽屉 / 详情区域

- 当前原始需求池并没有单独的详情抽屉，主交互是“卡片/表格内联编辑 + Accordion 就地展开详情”。
- 因此 AI 判级解释信息的主承载区应优先落在现有展开详情区 `detail-grid` 中，而不是新造一套大抽屉结构。
- 需要考虑展示：
  - 当前优先级；
  - 最近一次判定来源；
  - 规则版本；
  - 判定时间；
  - 判定摘要；
  - 命中规则摘要；
  - 人工修正记录。

### 2.3 新建原始需求入口

- 若系统希望在“提交后自动判级”，则新建原始需求后的处理链路会受到影响。
- 需要定义 AI 判级触发时机，是提交后异步执行，还是进入待评估状态后手动触发 / 自动触发。
- 当前系统中“新建原始需求”主要通过原始需求池页内的弹窗完成，而不是独立复杂提单页。
- 因此本期更适合在现有创建弹窗中补强关键上下文字段和提交后的状态反馈，而不建议将入口重心迁移为新独立页面。

### 2.4 优先级视图

- 该页面本身不是本期主改造目标，但会消费最终优先级结果。
- 本期优先级仍由单一字段承载，AI 自动判定和人工修正共同作用于同一个优先级字段，因此优先级视图无需处理双轨展示问题。

### 2.5 系统配置 / 规则维护入口

- 当前系统已存在 `/settings` 页面，并采用顶部 Tab 切换组织配置能力。
- 因此“优先级规则维护”与“AI 模型接入配置”更适合并入现有 `SettingsView.vue`，作为新的 Tab 或现有 Tab 下的子配置区，而不是另起一套独立导航页面。

## 3. 当前已知系统约束与依赖 (Constraints & Dependencies)

### 3.1 规则可编辑是核心前提

- 用户会提供一份文字版优先级定级规则。
- 规则内容需要支持后续持续编辑，因此不适合把判断逻辑硬编码在代码中。
- 规则应作为配置数据存储，而非散落在前端常量或后端 if/else 中。

### 3.2 AI 方案需要兼顾可维护性

- 需求重点不是“能否调模型”，而是“如何长期维护规则并稳定输出结果”。
- 设计上需要将“业务规则文本”和“模型控制提示词”分层管理。
- 后续应支持替换模型、调整提示词，而不破坏业务规则主体。

### 3.3 最终优先级仍需人工兜底

- 本期采用单一优先级字段模型，AI 自动判定结果直接回写该字段。
- 系统仍需保留人工修正入口，以应对规则灰区、上下文缺失、模型误判或业务临时策略变化。
- 人工修改后，系统记录操作日志和判定来源变更轨迹，但不额外拆分独立的“AI建议值 / 人工最终值”字段。

### 3.4 需要结构化输出与审计信息

- 由于模型输出天然不稳定，系统应要求 AI 返回结构化结果，而不是仅返回自由文本。
- 至少需要沉淀以下信息：
  - AI 生成并写入的优先级结果；
  - 判定理由摘要；
  - 命中规则；
  - 规则版本；
  - 判定时间；
  - 可选的置信度或异常标识；
  - 判定来源或最近一次覆写来源（AI / 人工）。

### 3.5 需要考虑工程层兜底

- AI 调用可能存在超时、失败、返回格式异常、成本波动等问题。
- 因此需要设计降级路径，例如：
  - 默认回退为“待人工判定”；
  - 给出默认 P2 并标明未成功完成 AI 判断；
  - 支持人工再次触发重判。

## 4. 当前阶段的设计假设 (Working Assumptions)

- 本期改造范围仅限“原始需求优先级自动判定”。
- 现有系统的原始需求数据结构已经有优先级字段，但尚无规则版本、判定摘要、判定来源等附加信息。
- 本期优先采用“规则文本 + 提示词模板 + 结构化输出”的低成本方案，而不是训练专用模型。
- 规则维护能力需要作为产品设计的一部分纳入，而不是仅在技术实现层面临时拼接。

## 5. 待你确认的关键点 (Need Confirmation)

- AI 判级触发时机是否默认为“原始需求提交后自动异步执行”？
- 规则维护权限是否只开放给管理员 / 产品负责人？
- 当 AI 与人工判断不一致时，是否需要强制填写人工修正原因？
