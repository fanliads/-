# 概念设计与方案选型 (Concept Design)

## 1. 设计目标

本期需要解决的问题不是单纯“接一个大模型接口”，而是要设计一套长期可维护的优先级自动判定机制。该机制需要同时满足以下目标：

- 支持基于文字版优先级规则进行自动判定；
- 规则可编辑，且修改后无需开发改代码；
- AI 与人工共同维护同一个优先级字段；
- 判定结果尽量稳定、可解释、可审计；
- 工程实现成本可控，适合在本期快速落地。

## 2. 方案候选

### 方案 A：纯提示词直判

#### 核心思路

- 将原始需求内容和当前生效的优先级规则文本直接拼接进提示词；
- 调用大模型，让模型直接返回优先级和判定原因；
- 系统把结果写入原始需求的 `priority` 字段，并记录判定摘要。

#### 优点

- 实现最快，开发成本最低。
- 最符合“规则是文字版”的原始输入形态，不需要先做复杂结构化。
- 当规则经常变化时，维护门槛低，业务人员容易理解。

#### 缺点

- 规则完全依赖模型理解，稳定性容易受提示词写法影响。
- 当规则文本越来越长、越来越复杂时，模型可能出现理解偏差。
- 可解释性有限，虽然可以要求模型返回理由，但命中依据未必足够稳定。
- 后续如果要做更细的监控、命中统计、局部规则调优，会比较被动。

#### 适用判断

- 适合 PoC 或极快试点。
- 不适合直接作为中长期主方案。

### 方案 B：规则文本可编辑 + 系统提示词模板约束 + 结构化输出

#### 核心思路

- 业务侧维护一份可编辑的优先级规则文本；
- 系统内置固定的提示词模板，负责约束模型角色、判定步骤和输出格式；
- 每次判级时，将“原始需求信息 + 当前规则文本 + 固定提示词模板”一起发给模型；
- 要求模型输出标准 JSON，例如：
  - `priority`
  - `reason`
  - `matched_rules`
  - `confidence`
  - `risk_flag`
- 系统校验返回结构，通过后回写同一个 `priority` 字段，并保存判定元数据。

#### 优点

- 保留规则文本可编辑能力，满足业务维护诉求。
- 通过固定模板和结构化输出，明显优于纯提示词直判的稳定性。
- 规则和系统控制逻辑分层，后续可替换模型或优化模板，不需要重写业务规则。
- 可解释性、审计性、失败兜底都更容易设计。
- 本期实现成本适中，风险可控。

#### 缺点

- 仍然依赖模型对自然语言规则的理解，不是完全确定性引擎。
- 需要额外设计输出校验、重试、降级逻辑。
- 当规则非常复杂时，可能仍需对规则文本做写法约束，否则模型效果波动。

#### 适用判断

- 适合本期落地。
- 兼顾速度、可维护性和后续扩展性，是当前最平衡的方案。

### 方案 C：代码规则引擎优先，AI 只处理模糊判断

#### 核心思路

- 先把优先级规则拆成结构化条件，配置成规则表或代码规则引擎；
- 能确定命中的规则直接由系统算出优先级；
- 只有无法命中或描述模糊时，才调用大模型补充判断。

#### 优点

- 稳定性最高，命中逻辑完全可控。
- 便于做规则命中统计、回归测试和版本管理。
- 对高频、清晰规则非常有效。

#### 缺点

- 前提是必须先把文字规则重构成结构化规则模型，设计和实施成本高。
- 规则可编辑性会下降，业务方通常无法直接维护，需要产品/开发转译。
- 本期目标强调“文字规则可编辑”，该方案与当前诉求不完全一致。
- 容易演变为一个小型规则引擎项目，超出本期范围。

#### 适用判断

- 适合作为后续演进方向。
- 不适合作为本期首发方案。

## 3. 方案对比

| 维度 | 方案 A：纯提示词直判 | 方案 B：规则文本 + 模板 + 结构化输出 | 方案 C：代码规则引擎优先 |
|---|---|---|---|
| 落地速度 | 高 | 中高 | 低 |
| 规则可编辑性 | 高 | 高 | 中低 |
| 稳定性 | 中低 | 中高 | 高 |
| 可解释性 | 中 | 中高 | 高 |
| 工程复杂度 | 低 | 中 | 高 |
| 后续扩展性 | 中 | 高 | 高 |
| 与本期目标匹配度 | 中 | 高 | 中 |

## 4. 推荐方案

### 选型结论

本期推荐采用 **方案 B：规则文本可编辑 + 系统提示词模板约束 + 结构化输出**。

### 推荐原因

- 它最符合当前需求的核心约束：规则必须是文字版、可编辑、可维护。
- 相比方案 A，它不是简单把规则扔给模型，而是增加了系统层约束，能显著提升结果稳定性。
- 相比方案 C，它不会把本期工作膨胀成规则引擎建设，能更快落地。
- 它为后续升级保留了空间：当积累更多案例后，可以再演进为“规则引擎 + AI补判”的混合模式。

## 5. 推荐落地形态

### 5.1 规则层

- 提供一个“优先级规则配置”入口，业务方维护自然语言规则文本。
- 规则以版本化配置存储，至少包含：
  - 规则名称
  - 规则内容
  - 是否生效
  - 更新时间
  - 更新人

### 5.2 模型调用层

- 系统维护一份固定提示词模板，不开放给普通业务用户直接编辑。
- 模板负责定义：
  - 你是优先级判定助手；
  - 你必须严格依据给定规则判断；
  - 你只能输出指定 JSON；
  - 若信息不足，返回固定降级标识。

### 5.3 输出处理层

- 模型返回结果后，系统执行：
  - JSON 格式校验；
  - 优先级值合法性校验；
  - 失败重试；
  - 降级兜底。
- 校验成功后，直接回写原始需求的 `priority` 字段。
- 同时记录：
  - 判定来源；
  - 规则版本；
  - 判定摘要；
  - 判定时间；
  - 人工后续修正记录。

### 5.4 人工协同层

- 原始需求详情中展示当前优先级和最近一次判定来源。
- AI 自动写入后，人工仍可直接改写同一字段。
- 人工改写后记录操作日志，并覆盖当前优先级值。

## 6. 演进建议

- 第一阶段：采用方案 B 快速上线，重点验证规则文本写法和模型输出稳定性。
- 第二阶段：沉淀误判案例，优化提示词模板和规则编写规范。
- 第三阶段：把高确定性规则逐步抽离成结构化规则，实现“规则引擎 + AI补判”的混合模式。
