# 发布上线 SOP

## 目标

规范需求管理系统的版本发布流程，确保每次上线都有明确版本、明确执行人、明确回滚路径。

## 适用范围

- 测试环境发布
- 预发环境发布
- 生产环境发布

## 角色分工

- 开发负责人
  - 完成功能开发、自测、提交代码
  - 确认本次发布内容和版本号
- 部署执行人或部署 agent
  - 只从代码仓库拉取指定版本
  - 执行构建、部署、重启和状态检查
- 验收负责人
  - 验证核心业务功能
  - 确认是否发布成功

## 发布原则

1. 不直接部署本地目录。
2. 只部署代码仓库中的明确版本。
3. 生产环境优先按 tag 发布，不按临时分支发布。
4. 发布失败时优先恢复服务可用性。
5. 不允许在服务器手改业务代码作为正式发布手段。

## 版本规范

- 正式版本使用 tag，例如：
  - `v1.0.0`
  - `v1.0.1`
  - `v1.1.0`
- 小写 `v` 作为统一前缀。
- 一个 tag 只对应一份代码，不重复覆盖。

## 发布前检查清单

### 代码检查

- 功能代码已合并到目标分支
- 本次发布代码已 push 到远程仓库
- 已创建发布 tag
- 发布说明已填写

### 配置检查

- 生产环境 `.env.production` 已配置
- 数据库连接信息已确认
- `JWT_SECRET` 已确认
- 第三方接口配置已确认

### 环境检查

- 服务器状态正常
- Docker 或运行时服务正常
- MySQL 正常
- Redis 正常
- 磁盘空间充足

### 数据检查

- 是否包含数据库脚本变更
- 是否需要备份数据库
- 是否需要灰度或错峰发布

## 标准发布流程

### 1. 确认版本

明确本次上线版本，例如：

```text
v1.0.0
```

### 2. 备份

如涉及数据库或高风险改动，先做备份。

### 3. 拉取版本

部署执行人或部署 agent 必须拉取指定 tag，不使用本地未提交代码。

示例：

```bash
git fetch --all --tags
git checkout tags/v1.0.0
```

### 4. 执行部署

按项目部署脚本执行构建和启动。

示例：

```bash
bash deploy/deploy.sh deploy tag v1.0.0
```

### 5. 服务检查

至少确认：

- 容器或服务已启动
- 日志无明显致命错误
- 健康检查通过

### 6. 业务验收

至少验证：

- 登录是否正常
- 核心页面是否正常打开
- 本次改动涉及流程是否正常
- 关键接口是否返回正常

### 7. 发布记录

记录以下信息：

- 发布时间
- 发布人
- 发布版本
- 发布结果
- 是否回滚

## 发布失败处理

- 如果是环境配置问题，先修复环境，再继续发布
- 如果是代码问题，优先回滚到上一个稳定 tag
- 不在服务器直接改业务代码作为正式修复
