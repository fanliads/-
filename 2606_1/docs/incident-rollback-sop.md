# 线上故障与回滚 SOP

## 目标

当线上发布后出现故障、异常或严重缺陷时，快速判断影响、控制范围、恢复服务，并为后续修复提供统一流程。

## 总原则

1. 先恢复可用性，再讨论优化。
2. 先判断是否需要回滚，不要第一时间在服务器改代码。
3. 所有修复都应最终回到代码仓库。
4. 已发布版本不直接覆盖，修复通过新版本重新发布。

## 故障分级

### 一级故障

影响核心业务，必须优先回滚。

示例：

- 无法登录
- 无法提交需求
- 核心页面大面积报错
- 服务无法启动
- 大量 500 错误

### 二级故障

影响部分功能，但核心业务仍可使用。

示例：

- 某个次要页面样式异常
- 某个非核心功能报错
- 某个入口操作失败，但存在替代路径

### 三级故障

影响较小，可记录后进入后续修复。

示例：

- 文案错误
- 轻微显示问题
- 个别边角交互不一致

## 故障处理流程

### 1. 先确认当前线上版本

明确当前部署版本，例如：

```text
v1.0.0
```

### 2. 判断影响等级

判断是否影响核心业务、是否影响大范围用户、是否造成数据风险。

### 3. 决策

- 一级故障：优先回滚
- 二级故障：评估是否临时止血后发布补丁
- 三级故障：记录问题并进入后续版本

### 4. 保留现场

至少保留：

- 错误日志
- 报错页面截图
- 请求参数
- 接口响应
- 发布时间和版本号

### 5. 回滚或修复

#### 回滚策略

回滚到上一个稳定 tag，例如：

```bash
git fetch --all --tags
git checkout tags/v0.9.9
```

或使用项目脚本：

```bash
bash deploy/deploy.sh rollback v0.9.9
```

#### 修复策略

- 不直接改服务器业务代码
- 在本地修复
- 提交到仓库
- 创建补丁版本，例如 `v1.0.1`
- 重新发布

### 6. 回滚后验证

至少验证：

- 服务是否恢复
- 核心功能是否恢复
- 日志中是否仍有明显异常

### 7. 记录与复盘

记录以下内容：

- 故障时间
- 影响范围
- 当前版本
- 原因分析
- 处理动作
- 回滚版本
- 修复版本
- 防再发措施

## 不允许的处理方式

- 直接在线上服务器修改业务源码并长期保留
- 用同一个 tag 覆盖不同代码
- 不留日志就反复重启服务
- 未确认影响范围就盲目发布补丁

## 版本修复建议

如果线上版本 `v1.0.0` 出现问题：

- 小修复：发布 `v1.0.1`
- 新功能修复：发布 `v1.1.0`
- 重大结构调整：发布 `v2.0.0`

不要重新覆盖 `v1.0.0`。
