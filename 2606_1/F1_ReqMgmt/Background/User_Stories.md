# 用户故事 (User Stories)

## 核心用户故事

### 故事 1：商务发起原始需求单
**作为** 商务人员，
**我希望** 能在系统中快速录入一个原始需求单，只需填写标题、描述和关联项目即可，
**以便于** 将客户的原始诉求正式提交给产品团队，无需填写复杂的技术字段。

**验收标准 (Acceptance Criteria):**
1. 点击「新建需求」后，表单仅展示精简字段：原始需求标题、原始需求描述、关联项目、需求单链接、附件上传。
2. 不展示优先级、业务线、产品模块、工作量估算等技术字段，降低填写门槛。
3. 提交后，原始需求单自动进入「待项目经理评估」状态，生成唯一编号（如 RAW-2605-001）。
4. 商务可在个人中心查看自己提交的所有原始需求单及其当前处理进度（如「待评估」「已拆分」「已拒绝」）。

---

### 故事 2：项目经理评估需求是否承接
**作为** 项目经理，
**我希望** 能在「我的待办」中查看待评估的需求单，并快速做出承接/拒绝/挂起的决策，
**以便于** 把控项目边界，避免资源浪费在不合理的方向上。

**验收标准 (Acceptance Criteria):**
1. 「我的待办」列表默认筛选出状态为「待项目经理评估」且指派给我的需求单。
2. 点击卡片后，侧滑抽屉顶部固定显示【承接】、【拒绝】、【挂起】三个操作按钮。
3. 选择【拒绝】或【挂起】时，必须填写原因，系统自动通知需求发起方。
4. 选择【承接】后，系统弹出「是否存在非功能性需求」的二次确认：若选择「是」，需求状态变更为「安排项目组研发处理」；若选择「否」，需求状态变更为「待产品总监判定」，并同步进入产品总监的待办。

---

### 故事 3：产品总监判定需求范围与类型
**作为** 产品总监，
**我希望** 能在需求池中查看待判定的需求，并区分其为定制化项目或标准产品需求，
**以便于** 从战略层面把控产品方向，避免标准产品被定制化需求侵蚀。

**验收标准 (Acceptance Criteria):**
1. 需求池视图支持按状态「待产品总监判定」筛选，卡片上显示需求来源和项目经理的承接结论。
2. 点击卡片后，详情页顶部显示【判定为定制化项目】和【判定为标准产品需求】两个操作。
3. 选择【判定为定制化项目】后，需求状态回退至「待项目经理承接评估」，并附带总监的定制化说明。
4. 选择【判定为标准产品需求】后，弹出「是否可实现」确认：若选择「否」，需求变更为「已拒绝」并记录原因；若选择「是」，进入需求拆分流程。

---

### 故事 4：项目经理补充粗糙的原始需求内容
**作为** 项目经理，
**我希望** 在评估商务提交的原始需求时，若发现内容过于粗糙或缺少关键信息，能够直接补充内容并记录补充原因，
**以便于** 下游的产品总监在判定时拥有更完整的上下文，减少来回沟通成本。

**验收标准 (Acceptance Criteria):**
1. 在原始需求单详情页，项目经理和产品总监均可见【补充内容】按钮。
2. 点击后弹出补充弹窗，要求选择补充类型（客户背景补充 / 技术细节补充 / 需求范围澄清 / 其他）并填写补充内容。
3. 补充内容提交后，以时间轴形式追加展示在原始需求单的概览页中，与原始描述视觉区分（如左侧带蓝色边框）。
4. 每条补充记录显示补充人头像、补充时间和补充类型标签。
5. 补充内容提交后，系统自动通知原始需求的提出人（如商务），告知其需求已被补充。

---

### 故事 5：产品总监拆分原始需求单为产品需求
**作为** 产品总监，
**我希望** 在判定原始需求可实现后，能够将一个原始需求单拆分为多个正式的产品需求，并分配给不同业务线，
**以便于** 复杂诉求能被不同团队并行处理，同时保留原始需求作为追溯根因。

**验收标准 (Acceptance Criteria):**
1. 在原始需求单详情页提供【拆分需求】功能，弹出拆分弹窗。
2. 弹窗左侧固定展示原始需求信息（只读），右侧为拆分操作区。
3. 支持两种拆分模式：【拆分为 N 个产品需求】或【无需拆分，直接转为 1 个产品需求】。
4. 每个产品需求条目可独立填写：标题、描述（支持从原始描述一键复制粘贴并修改）、业务线、产品模块、优先级、期望上线时间、工作量估算、价值量化评估。
5. 产品需求创建后生成独立编号（如 PRD-2605-001），自动关联父原始需求单编号。
6. 拆分完成后，原始需求单状态变更为「已拆分」，所有产品需求状态为「待产品组长过滤」，并通知对应产品组长。

---

### 故事 6：产品组长过滤与分发需求
**作为** 产品组长，
**我希望** 能在产品需求池中评估和过滤进入的产品需求，将合理需求按产品图谱分发给对应业务线产品经理，
**以便于** 把控需求质量，确保下游产品经理接收到的需求是清晰且合理的。

**验收标准 (Acceptance Criteria):**
1. 产品需求池支持按「待产品组长过滤」状态筛选，按业务线分组展示。
2. 点击卡片后，详情页顶部显示【通过并分发】、【挂起】两个操作。
3. 选择【挂起】时必须填写原因，系统自动通知上游（产品总监）。
4. 选择【通过并分发】时，弹出业务线产品经理选择器，选定后需求状态变更为「待产品经理接手」，并通知对应产品经理。

---

### 故事 7：产品经理进行需求调研与迭代排期
**作为** 业务线产品经理，
**我希望** 接手需求后，能判断是否需要调研，并将确认后的需求纳入迭代计划，
**以便于** 在开发前有充足的准备，避免返工。

**验收标准 (Acceptance Criteria):**
1. 「我的待办」中状态为「待产品经理接手」的需求卡片显示「接手」按钮，点击后状态变更为「待办需求」。
2. 需求详情页提供【开始调研】按钮，点击后状态变更为「调研中」，并可记录调研结论。
3. 调研完成后，提供【纳入迭代计划】操作，弹出迭代选择器（支持新建迭代）。
4. 纳入迭代后，需求状态变更为「设计中」，并出现在对应迭代的看板中。

---

### 故事 8：产品经理进行设计确认与研发推进
**作为** 业务线产品经理，
**我希望** 在需求设计完成后，能根据是否需要用户确认走不同的确认分支，最终推进至研发上线，
**以便于** 确保设计方案符合用户预期，同时减少不必要的确认环节。

**验收标准 (Acceptance Criteria):**
1. 需求状态为「设计中」时，详情页显示【提交确认】按钮，弹出确认方式选择：「需要用户确认」或「内部确认」。
2. 选择「需要用户确认」后，需求状态变更为「等待用户确认」，并生成确认链接/通知发送给外部用户。
3. 用户确认通过后，状态自动变更为「研发中」；若用户拒绝，则退回「设计中」并记录反馈。
4. 选择「内部确认」后，由产品组长或指定角色在系统内一键确认，确认后状态变更为「研发中」。
5. 研发完成后，状态变更为「测试中」；测试通过后，状态变更为「已上线」。

---

### 故事 9：团队成员在看板中处理待办需求
**作为** 研发工程师，
**我希望** 能在看板视图中查看分配给自己的需求卡片，拖拽更新状态并添加进度说明，
**以便于** 高效管理日常开发任务，保持团队信息同步。

**验收标准 (Acceptance Criteria):**
1. 看板视图支持按「我的任务」筛选，仅展示负责人为自己的需求卡片。
2. 需求卡片表面显示标题、优先级标签、截止日期倒计时（逾期标红）。
3. 支持拖拽卡片在看板列之间移动（如从「研发中」拖到「测试中」），拖拽时系统弹出确认框并允许填写进度说明。
4. 拖拽后，系统自动通知下游角色（如从「测试中」拖到「已上线」时通知产品经理）。

---

### 故事 10：管理层查看需求大盘与价值回归评估
**作为** 部门负责人，
**我希望** 能在数据看板中查看当前迭代的需求完成率、各业务线需求分布、以及已上线需求的价值评估结果，
**以便于** 做出资源调配和产品方向决策。

**验收标准 (Acceptance Criteria):**
1. 数据分析页面提供「需求大盘」仪表盘，展示：本月新增需求数、各状态需求分布饼图、各业务线需求吞吐量柱状图。
2. 支持按时间范围（本周/本月/本季度）和业务线维度筛选。
3. 已上线需求卡片显示「发起价值回归评估」按钮，点击后弹出评分表单（1-5星）和评论框。
4. 价值评估结果汇总在数据看板的「价值回归」模块，展示平均评分和评价数量。
