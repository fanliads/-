# 需求背景文档

## 1. 需求概述 (Overview)

本需求旨在设计一款面向产品团队日常需求管理的应用，解决产品团队在日常工作中需求来源分散、状态不透明、评审流程缺失、排期信息不同步等核心痛点。通过统一的需求管理平台，实现从需求提出到上线归档的全生命周期管理。

### 1.1 目标用户 (Target User)

- **需求发起方**: 市场部（负责编写并录入需求单）、实施/客服/商务（负责提出建议和问题）。
- **项目经理及项目组**: 负责评估需求是否承接，判断是否存在非功能性需求，安排研发处理与项目排期。
- **产品总监**: 负责判断需求范围和类型，区分定制化项目与标准产品需求，把控可实现性。
- **产品组长**: 负责接手产品需求池，进行需求评估及过滤，按产品图谱分发至对应业务线产品经理。
- **产品经理**: 负责需求调研、纳入迭代计划、产品设计、用户确认、研发推进及上线后的价值回归评估。
- **参与用户**: 研发工程师、设计师、测试工程师（负责需求参与、任务执行、状态反馈）。
- **管理层**: 部门负责人（负责需求大盘查看、资源决策）。

### 1.2 核心价值 (Core Value)

- **统一入口**: 将所有需求汇聚到单一平台，避免散落在文档、IM、邮件中。
- **流程透明**: 五角色协作流程可视，需求状态流转实时同步，团队成员随时了解当前进展。
- **分层过滤**: 通过项目经理承接评估、产品总监范围判定、产品组长合理性过滤，形成多层质量 gate。
- **高效流转**: 支持需求单拆分、业务线划分、自动分发，减少人工交接成本。
- **闭环管理**: 从需求录入到价值回归评估，形成完整的 PDCA 闭环。
- **数据沉淀**: 需求历史可追溯，支持复盘与知识库沉淀。

## 2. 详细功能需求 (Detailed Requirements)

### 2.1 核心功能

- **需求创建与管理**: 支持创建需求单，包含标题、描述、来源渠道（市场/实施/客服/商务）、优先级、标签、截止日期、负责人等字段。
- **状态流转引擎**: 支持自定义或预设状态流。原始需求单的当前状态包含：空、评估中、待设计、已设计、开发中、已上线、拒绝、挂起、验收不通过。所有字段支持编辑，编辑记录以 Timeline 形式保存并可展示。优先级定级支持 AI 自动判定（基于可配置的文字化规则）+ 人工调整。
- **评审与过滤**: 支持项目经理承接评估、产品总监可实现性判定、产品组长合理性过滤，每层评估均可记录结论与原因。
- **需求拆分与分发**: 支持将需求单拆分为多个产品需求，按业务线自动分发至对应产品经理。
- **排期与迭代关联**: 支持将需求关联到具体迭代版本，设置预计工时，查看排期冲突。
- **多视图支持**: 提供看板视图（Kanban）、列表视图（List）和日历视图（Calendar）三种方式查看需求。
- **需求关联**: 支持需求间建立关联关系，如前置依赖、父子需求、相关需求。
- **搜索与筛选**: 支持按状态、优先级、标签、负责人、业务线、迭代等多维度筛选与全文搜索。
- **通知与提醒**: 状态变更、评估结论、评论回复、截止日期临近时自动通知相关人员。
- **价值回归评估**: 需求上线后支持发起价值回归评估，沉淀价值评价数据。

## 3. 核心场景 (Core Scenarios)

### 3.1 市场部发起需求并推动流转

- **场景描述**: 市场部或一线人员（实施/客服/商务）在日常工作中识别到新需求或问题，需要录入系统并推动落地。
- **典型流程**:
    1. 市场部编写并录入需求单，或实施/客服/商务提出建议和问题。
    2. 需求同步至项目经理侧，项目经理评估是否承接。
    3. 若不承接，则拒绝/挂起并通知需求发起方。
    4. 若承接，项目经理判断是否存在非功能性需求：若存在，则安排项目组直接研发处理；若不存在，则进入项目排期，并同步至产品组长。

### 3.2 产品总监进行需求范围与类型判定

- **场景描述**: 产品总监在需求池中对需求进行范围和类型判定，区分定制化项目与标准产品需求。
- **典型流程**:
    1. 产品总监查看待判定的需求单，判断需求范围和类型。
    2. 若判定为定制化项目，则回流至项目经理侧进行承接评估。
    3. 若判定为标准产品需求，则进一步判断需求是否可实现。
    4. 若不可实现，则拒绝该需求并记录原因。
    5. 若可实现，则将需求单拆分为多个产品需求，按业务线划分后进入需求池。

### 3.3 产品组长进行需求过滤与分发

- **场景描述**: 产品组长接手需求池中的产品需求，进行质量过滤并按产品图谱分发给对应业务线产品经理。
- **典型流程**:
    1. 产品组长（或指定产品经理）接手进入需求池的产品需求。
    2. 进行需求评估及过滤，识别不合理需求。
    3. 若存在不合理需求，则挂起并记录原因。
    4. 合理需求进入待办列表，按照产品图谱流转至对应业务线产品经理。

### 3.4 产品经理进行需求落地与闭环

- **场景描述**: 业务线产品经理接手需求后，完成调研、设计、研发推进及上线后的价值评估。
- **典型流程**:
    1. 对应业务线产品经理接手需求。
    2. 判断是否需要产品调研：若需要，则开展调研，完成后纳入迭代计划；若不需要，直接纳入迭代计划。
    3. 进入产品设计阶段，产出设计方案。
    4. 判断是否需要用户确认：若需要，则提交用户确认；用户未确认则返回设计阶段，用户确认后进入研发上线阶段。
    5. 若不需要用户确认，则在内部确认后直接进入研发上线阶段。
    6. 研发上线阶段验收通过后，判断是否发起价值回归评估。
    7. 若发起价值回归评估，则进入价值评价环节，完成闭环。

### 3.5 跨需求依赖与排期管理

- **场景描述**: 需求之间存在前置依赖关系，需要系统识别并预警，辅助排期决策。
- **典型流程**:
    1. 产品经理在创建或编辑需求时，设置前置依赖需求。
    2. 系统在看板和列表中显示依赖关系图标。
    3. 当前置依赖需求进度延迟或状态未按时推进时，系统自动向被依赖需求的负责人发送风险预警。
    4. 负责人根据预警及时协调资源或调整排期。
