# 系统现状调研 (System Survey)

## 1. 需求定位

本次需求为 **全新系统设计**，非现有系统迭代或模块改造。目标为打造一款面向产品团队日常需求管理的独立应用，覆盖从需求发起、评估、过滤、设计、研发到上线后价值回归评估的完整生命周期。

## 2. 原始系统核心逻辑 (Original System Core Logic)

- **现状**: 无现有系统。当前需求管理流程依赖线下文档、IM 工具、邮件或分散的表格进行管理。
- **痛点**: 需求来源分散、状态不透明、五角色协作流程无系统化支撑、缺乏分层过滤机制、排期信息不同步、历史数据难以追溯。

## 3. 受影响的现有页面 (Impacted Existing Pages)

- **现状**: 无现有页面。所有页面均需全新设计。
- **设计自由度**: 高。无需考虑向后兼容或现有 UI/交互约束。

## 4. 约束与依赖 (Constraints & Dependencies)

| 维度 | 约束/依赖描述 |
|------|--------------|
| **系统集成** | 暂不考虑与现有外部系统（SSO、组织架构、消息通知、文档管理、项目管理工具等）的集成，设计阶段按独立系统处理。后续可扩展。 |
| **用户体系** | 需内置独立的用户、角色与权限体系，覆盖市场、项目经理、产品总监、产品组长、产品经理五类核心角色。 |
| **部署方式** | 未明确指定，按 Web 应用设计，兼容主流浏览器。 |
| **技术栈** | 未指定，前端设计按通用 Web 标准输出。 |
| **数据迁移** | 无历史数据迁移需求。 |
| **合规要求** | 未指定特殊合规要求。 |

## 5. 关键设计假设 (Key Design Assumptions)

1. 系统内置完整的 RBAC 权限模型，各角色只能看到和处理自己职责范围内的需求。
2. 需求状态机需支持用户自定义配置，以适配不同组织的流程差异。
3. 通知机制初期以内置站内通知为主，后续可扩展为邮件/IM 推送。
4. 价值回归评估模块为轻量级打分与评论机制，非复杂 BI 分析系统。
