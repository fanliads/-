# 概念设计推演 (Concept Design)

## 背景

基于 `Requirement_Background.md` 和 `System_Survey.md`，本系统需支撑五角色（市场/项目经理/产品总监/产品组长/产品经理）的协作，覆盖从需求发起到价值回归的完整生命周期。核心设计挑战在于：**如何在严格的审批过滤流程与高效的协作体验之间取得平衡**。

---

## 方案 A：审批流驱动（Workflow-First）

### 设计思路
以「工作流节点」为核心组织方式。每个角色拥有独立的「待办工作台」，需求按预设流程节点（如：待评估→待判定→待过滤→待调研→设计中→研发中→已上线）线性流转。每个节点变更需由对应角色主动审批或确认后才能推进到下一阶段。

### 页面架构
- **全局导航**: 我的待办 | 需求管理 | 迭代管理 | 数据分析
- **核心视图**: 各角色登录后默认进入「我的待办」，展示当前需要本人处理的需求清单。
- **需求详情**: 以表单形式呈现需求字段，底部固定「通过 / 驳回 / 转交」操作栏。
- **流转记录**: 独立时间轴页面，记录每一次审批结论与原因。

### 优点 (Pros)
1. **流程严谨**: 每个 gate 都有明确的责任人和审批动作，符合五角色分层过滤的组织要求。
2. **权限清晰**: 角色与节点强绑定，不易出现越权操作。
3. **审计友好**: 每个流转节点都有明确的通过/驳回记录，便于事后审计。

### 缺点 (Cons)
1. **上下文割裂**: 用户无法一眼看到需求的整体进展，需要在「我的待办」和「需求详情」之间频繁切换。
2. **操作路径长**: 完成一次审批通常需要「进入待办 → 打开详情 → 阅读内容 → 点击通过 → 返回列表」五步。
3. **协作感弱**: 评论、@提及等协作功能容易被淹没在审批流程中。
4. **灵活性差**: 若流程节点需要调整，系统改动成本较高。

---

## 方案 B：看板协作驱动（Board-First）

### 设计思路
以「看板（Kanban）」为核心组织方式。需求以卡片形式在看板列中呈现，看板列直接映射需求状态（如：待评估→待判定→待过滤→待办需求→调研中→设计中→研发中→测试中→已上线→已关闭）。所有角色在同一视图中协作，状态流转通过拖拽卡片完成。审批 gate 以卡片内的「评审记录」或「阻断标记」形式存在，既保留流程约束，又不牺牲协作效率。

### 页面架构
- **全局导航**: 看板 | 列表 | 日历 | 需求池 | 迭代 | 我的待办
- **核心视图**: 看板视图作为默认首页，支持按角色、业务线、迭代等多维度筛选。
- **需求卡片**: 卡片表面展示关键信息（标题、优先级、负责人、标签、截止日期）， hover 或点击展开快速预览面板。
- **需求详情**: 侧滑抽屉（Side Drawer）形式，不离开当前视图即可查看完整信息、发表评论、执行审批。
- **流转记录**: 详情页内嵌时间轴，与评论流合并呈现。

### 优点 (Pros)
1. **全局可视**: 所有需求的状态分布一目了然，管理者可以快速识别瓶颈。
2. **操作高效**: 拖拽即可变更状态，审批可在侧滑抽屉中一键完成，无需页面跳转。
3. **协作自然**: 评论、@提及、附件与需求卡片紧密绑定，讨论有上下文。
4. **灵活扩展**: 看板列可由管理员自定义配置，适配不同团队的流程差异。

### 缺点 (Cons)
1. **审批 gate 弱化风险**: 若设计不当，用户可能忽略卡片上的待审批标记，导致流程跳过。
2. **信息密度挑战**: 当需求数量庞大时，看板可能变得拥挤，需要配合良好的筛选与折叠机制。
3. **学习成本**: 对于习惯传统审批系统的用户，看板操作需要一定的适应期。

---

## 最终方案选择

**选择：方案 B（看板协作驱动）为主，融入方案 A 的审批 gate 机制作为辅助。**

### 选择理由
1. **符合现代产品团队习惯**: 产品、研发团队对看板（Jira/Trello/飞书项目）已有高度认知，降低学习成本。
2. **解决核心痛点**: 用户流程图最大的痛点是「信息分散、状态不透明」，看板视图天然解决了这一问题。
3. **兼顾流程严谨性**: 通过「状态流转规则引擎」+「待审批阻断提示」+「流转日志」三重机制，确保五角色的审批 gate 不会弱化。
4. **可扩展性强**: 看板列、状态流、字段均可配置，为后续适配不同组织留下空间。

### 关键设计决策
- **默认首页**: 看板视图（可按角色记住上次使用的视图偏好）。
- **审批体验**: 当需求进入需要当前用户审批的状态时，卡片角标显示「待我处理」红点；点击卡片后，侧滑抽屉顶部固定显示审批操作栏（通过 / 驳回 / 转交 / 挂起）。
- **多视图并存**: 看板为核心，列表视图用于批量操作，日历视图用于排期审视，满足不同场景。
- **需求拆分**: 在需求详情中提供「拆分为产品需求」功能，拆分后子需求自动进入对应业务线的看板列。
